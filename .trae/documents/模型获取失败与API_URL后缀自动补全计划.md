# 模型获取失败与 API URL 后缀自动补全计划

## 问题背景
在设置页 API 配置区域，供应商可填写自定义 API URL。当前若仅填写域名或基础路径（例如 `https://hajimi.zabc.net/`），点击“获取模型列表”可能失败，因为后端实际调用模型接口通常需要完整后缀（如 `/v1/chat/completions`）。

## 目标
- 在不改变现有使用方式的前提下，提升 API URL 输入容错性。
- 对缺少标准后缀的 URL 做检测与自动补全，减少“模型不能获取”的错误。
- 保持前后端配置数据一致，避免重复补全或错误拼接。

## 实施范围
- 前端：`frontend/src/pages/SettingsPage.tsx`
- 后端：`backend/billing/pricing_manager.py`
- 验证：前端类型检查与后端相关测试/接口冒烟

## 实施步骤

### 步骤 1：梳理并定义 URL 归一化规则
1. 识别当前 provider 维度可接受的完整端点格式：
   - OpenAI/DeepSeek/Moonshot/Alibaba/Zhipu：默认补全 `/v1/chat/completions`
   - Anthropic：默认补全 `/v1/messages`
   - Google：默认补全 `/v1beta/models`
2. 约定补全触发条件：
   - 用户输入为空：不补全
   - 已包含已知后缀：不补全
   - 仅为域名/基础路径：自动补全
3. 约定路径拼接规则：
   - 去除末尾多余 `/`
   - 保证补全后仅一个分隔 `/`

### 步骤 2：前端输入侧补全（即时体验）
1. 在设置页新增纯函数：
   - `normalizeProviderApiEndpoint(provider, apiEndpoint)`
2. 在以下保存动作前统一调用：
   - 新增供应商 `handleCreateProvider`
   - 保存供应商配置 `handleSaveProviderConfig`
3. 对“获取模型列表”按钮交互增加提示：
   - 若 URL 被自动补全，给出一次性成功提示（例如“已自动补全 API URL 后缀”）
4. 保持旧数据兼容：
   - 已配置完整后缀的历史 URL 不变

### 步骤 3：后端持久化侧兜底补全（数据一致性）
1. 在 `PricingManager._normalize_configuration_payload` 中对 `api_endpoint` 增加 provider 感知的归一化逻辑。
2. 归一化逻辑与前端规则一致，防止前后不一致。
3. 确保后端作为最终兜底，即使前端未补全也能存成可用地址。

### 步骤 4：获取模型链路回归验证
1. 针对以下输入验证“获取模型列表”成功率：
   - 仅域名（无路径）
   - 仅基础路径（如 `/v1`）
   - 已完整后缀 URL
2. 验证供应商切换和保存后回显：
   - 回显应为补全后的最终地址
3. 确认不会影响已有 provider 的 selected_models 流程。

### 步骤 5：质量验证
1. 前端执行：`npm run typecheck`
2. 后端执行相关测试（至少覆盖配置归一化路径）
3. 手工冒烟：设置页“获取模型列表”与“保存供应商配置”

## 验收标准
- 输入不完整 API URL 时，保存后能够自动补全为可调用端点。
- 点击“获取模型列表”不再因缺失后缀导致普遍失败。
- 已完整 URL 不会被错误改写。
- 前后端校验与类型检查通过。

## 风险与应对
- 风险：不同代理服务的路径规则与官方端点不一致。
  - 应对：仅在缺失路径时补全；若用户已给出明确路径则尊重用户输入。
- 风险：前后端重复补全导致双后缀。
  - 应对：先做“已包含后缀”检测，再执行补全。
