# Refactor Model Selection and Import Spec

## Why
当前系统在前端对话页面的模型选择（`select` 下拉框）与 API 配置中的模型列表（`section` 区域）存在状态不一致的情况。同时，当用户点击“获取模型列表”时，缺乏一个明确的“选择并导入”缓冲环节，导致拉取的所有模型直接混入配置中，管理成本高。为了提升体验，需要引入“拉取 -> 弹窗选择 -> 导入 -> 管理（含批量删除）”的规范工作流，并确保聊天页面的可用模型与配置页严格同步。

## What Changes
- **后端接口调整**：调整模型列表获取逻辑，使其仅返回远端模型列表而不自动覆盖本地配置；新增或完善模型导入（保存）与批量删除接口。
- **前端弹窗与导入机制**：点击“获取模型列表”成功后，弹出一个模型选择对话框，展示所有可用模型；用户支持多选并点击“导入”后，才真正写入本地配置。
- **配置页展示与批量删除**：在 API 配置界面仅展示已导入的模型，并增加“批量删除”按钮及二次确认弹窗，支持勾选并一次性移除不需要的模型。
- **全局状态同步**：重构模型数据的状态流转，确保 API 配置页发生导入/删除后，聊天界面的模型选择下拉框（`select`）能实时、准确地同步最新可用模型。
- **交互与容错体验**：在所有网络请求环节（获取、导入、删除）补充加载中（Loading）状态、错误提示（Toast/Message）及操作确认，保障异常网络下的交互稳定性。

## Impact
- Affected specs: 供应商配置模块、模型管理模块、聊天页面模型选择。
- Affected code:
  - `frontend/src/features/settings/SettingsPage.tsx` （模型展示、弹窗、批量删除逻辑）
  - `frontend/src/features/chat/ChatPage.tsx` （模型选择下拉框同步）
  - `frontend/src/shared/store/` （可能涉及状态管理的 store 更新）
  - `backend/billing/routers/billing.py` （后端模型获取与保存/删除接口）
  - `backend/billing/pricing_manager.py` （模型配置的增删改查逻辑）

## ADDED Requirements
### Requirement: 模型拉取与导入流程
系统 SHALL 提供一个清晰的模型拉取与导入缓冲工作流。

#### Scenario: 成功拉取并导入模型
- **WHEN** 用户在配置页点击“获取模型列表”
- **THEN** 系统展示加载状态，成功后弹出包含远端模型列表的对话框。
- **WHEN** 用户在对话框中勾选目标模型并点击“导入”
- **THEN** 系统保存这些模型，并在配置页的供应商区域展示这些已导入的模型。

### Requirement: 批量删除已导入模型
系统 SHALL 允许用户批量管理和移除不再需要的已导入模型。

#### Scenario: 批量删除模型
- **WHEN** 用户在配置页勾选多个已导入的模型并点击“批量删除”
- **THEN** 系统弹出二次确认框，用户确认后系统将其从配置中移除，并更新界面。

## MODIFIED Requirements
### Requirement: 聊天页模型状态同步
系统 SHALL 保证聊天页的模型下拉框严格与配置页已导入的模型列表保持一致。

#### Scenario: 状态实时同步
- **WHEN** 配置页发生模型的导入或删除操作
- **THEN** 聊天页的模型下拉框立即更新，仅显示当前处于启用/已导入状态的模型，避免出现“幽灵模型”或配置缺失。
