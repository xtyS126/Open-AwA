# 简化供应商配置界面 Spec

## Why
当前 API 配置界面仅支持单一供应商下拉选择，无法像用户示意图那样在左侧集中管理多个供应商，也不支持在配置完成后按供应商拉取模型列表并由用户自行勾选。这使得多供应商接入、模型启用管理和后续聊天模型选择体验都不够直观。

## What Changes
- 前端新增“供应商列表 + 右侧配置面板”的简化双栏布局
- 支持在左侧新增自定义供应商条目，并可选配置供应商图标
- 点击左侧供应商后，在右侧编辑该供应商的 API URL、API Key 与基础信息
- 配置完成后支持按当前供应商拉取模型列表，并通过复选框让用户自主选择启用的模型
- 将已选模型保存为该供应商的启用模型集合，供聊天模型选择等后续功能使用
- 后端补齐供应商配置与模型拉取/保存所需接口或数据结构
- 保持界面实现简单，避免照搬示意图中的复杂筛选、分组和状态装饰

## Impact
- Affected specs: 设置页 API 配置、模型管理、聊天模型来源
- Affected code:
  - `frontend/src/pages/SettingsPage.tsx`
  - `frontend/src/pages/SettingsPage.css`
  - `frontend/src/services/modelsApi.ts`
  - `backend/billing/routers/billing.py`
  - `backend/billing/pricing_manager.py`
  - 与供应商配置持久化相关的数据模型/Schema

## ADDED Requirements

### Requirement: 左侧供应商列表管理
系统 SHALL 在 API 配置区域提供一个左侧供应商列表，允许用户浏览、切换和新增供应商。

#### Scenario: 查看供应商列表
- **WHEN** 用户进入 API 配置界面
- **THEN** 页面左侧应显示已存在的供应商列表
- **AND** 每个供应商应至少显示名称
- **AND** 若已配置图标则显示图标，否则显示文字占位

#### Scenario: 新增供应商
- **WHEN** 用户点击“新增供应商”
- **THEN** 系统应创建一个新的待编辑供应商条目
- **AND** 用户可填写供应商名称
- **AND** 用户可选择是否上传或填写图标

#### Scenario: 切换供应商
- **WHEN** 用户点击左侧某个供应商条目
- **THEN** 右侧应加载该供应商的当前配置
- **AND** 当前选中的供应商应有清晰高亮状态

### Requirement: 供应商基础配置编辑
系统 SHALL 允许用户为单个供应商维护基础访问配置。

#### Scenario: 编辑 API 连接信息
- **WHEN** 用户在右侧配置面板修改 API URL 或 API Key
- **THEN** 页面应允许保存当前供应商配置
- **AND** 保存成功后应保留用户输入并反馈成功状态

#### Scenario: API Key 安全展示
- **WHEN** 页面显示已保存的 API Key 字段
- **THEN** 默认应使用密码输入框样式
- **AND** 不应在非必要位置明文暴露密钥

### Requirement: 按供应商获取模型列表
系统 SHALL 支持在供应商完成基础配置后获取该供应商可用模型列表。

#### Scenario: 获取模型列表成功
- **WHEN** 当前供应商已填写有效 API URL 与 API Key，且用户点击“获取模型”
- **THEN** 系统应请求后端拉取该供应商的模型列表
- **AND** 在界面中展示返回的模型项
- **AND** 展示方式应以简单列表为主，不强制复杂分组

#### Scenario: 获取模型列表失败
- **WHEN** 模型拉取失败
- **THEN** 页面应显示明确错误提示
- **AND** 不应清空用户已填写的基础配置

### Requirement: 模型复选启用
系统 SHALL 允许用户通过复选框选择当前供应商启用的模型。

#### Scenario: 勾选模型
- **WHEN** 模型列表加载完成
- **THEN** 每个模型项应提供复选框
- **AND** 用户可选择一个或多个模型
- **AND** 已选模型状态应可见

#### Scenario: 保存已选模型
- **WHEN** 用户点击保存供应商配置
- **THEN** 当前供应商的已选模型应一并保存
- **AND** 下次重新进入该供应商时应恢复已保存的勾选状态

### Requirement: 简化界面结构
系统 SHALL 以简洁实现完成该界面，避免不必要的复杂交互。

#### Scenario: 简化展示
- **WHEN** 用户使用该界面
- **THEN** 左侧仅保留供应商列表与新增入口
- **AND** 右侧仅保留基础配置区、模型获取区、模型勾选区和保存操作
- **AND** 不实现复杂筛选标签、层级折叠、过多状态徽标等非必要元素

## MODIFIED Requirements

### Requirement: 设置页 API 配置交互
现有 API 配置区域从“单个 Provider 下拉选择 + 单表单编辑”修改为“左侧供应商列表 + 右侧供应商详情配置”的交互方式。

#### Scenario: 从旧交互迁移到新布局
- **WHEN** 用户打开更新后的 API 配置界面
- **THEN** 应看到双栏布局而非单一 Provider 下拉框
- **AND** 现有配置数据应尽量映射到新的供应商项中显示

### Requirement: 模型来源展示
现有模型管理能力需要支持“按当前供应商实时获取模型”与“按已勾选结果持久化”的组合流程，而不仅是读取全局默认模型配置。

## REMOVED Requirements

### Requirement: API 配置仅支持固定供应商下拉选择
**Reason**: 该方式不适合多供应商并行管理，也无法匹配用户希望的左侧供应商入口交互。
**Migration**: 将已有固定供应商配置映射为默认供应商条目，并在新界面中继续编辑。
