## 2. 候选项目逐一评估

### 2.1 FlexSearch

| 维度 | 评估 |
|------|------|
| **仓库** | github.com/nextapps-de/flexsearch |
| **本地运行** | 纯JS，浏览器+Node.js双环境，零服务端依赖 |
| **索引性能** | 极快，声称"最快的JS搜索库"，采用优化的倒排索引 |
| **中文分词** | 支持CJK编码器（`encode: "balance"`），但有局限性；可通过自定义encoder集成jieba等分词器 |
| **排序算法** | 支持BM25等多级评分，可自定义排序函数 |
| **兼容性** | 纯JS，与React/Vue/Node.js完全兼容 |
| **构建体积** | ~15KB gzipped（基础版），可按需加载language pack |
| **维护状态** | v0.7.x稳定维护中，v1.0正在开发 |

**优势**: 速度极快，体积极小，API简洁，支持异步搜索和web worker
**劣势**: 中文分词需自定义encoder，文档较少，v1.0开发周期长

### 2.2 Orama (原Lyra)

| 维度 | 评估 |
|------|------|
| **仓库** | github.com/oramasearch/orama |
| **本地运行** | 纯TS/JS，零依赖，浏览器+Node.js |
| **索引性能** | 非常快，使用前缀树+倒排索引混合结构 |
| **中文分词** | 内置支持Unicode标准化，中文需要自定义tokenizer |
| **排序算法** | BM25 + 自定义评分，支持facets过滤 |
| **兼容性** | TypeScript原生，完美支持React/Vue |
| **构建体积** | ~20KB gzipped |
| **维护状态** | 活跃开发中，社区活跃，文档完善 |

**优势**: TypeScript原生，API现代且易用，零依赖，支持向量搜索(hybrid)
**劣势**: 中文支持需额外配置，相对于FlexSearch体积稍大

### 2.3 MiniSearch

| 维度 | 评估 |
|------|------|
| **仓库** | github.com/lucaong/minisearch |
| **本地运行** | 纯JS，浏览器+Node.js |
| **索引性能** | 良好，使用倒排索引+前缀搜索 |
| **中文分词** | 内置tokenizer支持Unicode，但中文需配置`tokenize`函数 |
| **排序算法** | TF-IDF + 字段权重 + 自定义评分 |
| **兼容性** | TypeScript类型定义完善，React/Vue兼容 |
| **构建体积** | ~10KB gzipped |
| **维护状态** | 稳定维护，文档优秀，示例丰富 |

**优势**: 极轻量，API设计优雅，文档完善，TypeScript支持好
**劣势**: 功能相对简单，不支持复杂查询语法，中文分词需自定义

### 2.4 Fuse.js

| 维度 | 评估 |
|------|------|
| **仓库** | github.com/krisk/fuse |
| **本地运行** | 纯JS，浏览器+Node.js |
| **索引性能** | 中等（基于逐项匹配，非倒排索引） |
| **中文分词** | 字符级匹配，天然支持中文（模糊匹配） |
| **排序算法** | 基于Bitap算法的模糊匹配评分 |
| **兼容性** | 通用JS库，与React/Vue兼容 |
| **构建体积** | ~5KB gzipped |
| **维护状态** | 稳定，但更新频率较低 |

**优势**: 极小体积，中文开箱即用，模糊搜索强，API简单
**劣势**: 非真正的全文搜索引擎，大数据集性能差，无索引持久化

### 2.5 Pagefind

| 维度 | 评估 |
|------|------|
| **仓库** | github.com/CloudCannon/pagefind |
| **本地运行** | Rust编译为WASM，浏览器端运行 |
| **索引性能** | 极快（Rust + WASM），索引构建在构建时完成 |
| **中文分词** | 支持Unicode分词，中文分词基于字符n-gram |
| **排序算法** | 基于页面权重+关键词匹配度 |
| **兼容性** | 框架无关，提供JS接口和Web Component |
| **构建体积** | WASM约100KB gzipped |
| **维护状态** | CloudCannon公司维护，活跃开发中 |

**优势**: 索引构建快，WASM性能好，专为静态站点设计，CLI工具完善
**劣势**: 主要面向静态站点场景，灵活性较低，体积较大

### 2.6 Tantivy

| 维度 | 评估 |
|------|------|
| **仓库** | github.com/quickwit-oss/tantivy |
| **本地运行** | Rust库，可通过Python绑定或WASM编译本地运行 |
| **索引性能** | 极快（Rust实现，类似Lucene） |
| **中文分词** | 支持，通过tantivy-jieba集成 |
| **排序算法** | BM25 + 自定义评分 |
| **兼容性** | Python绑定可用，WASM前端可用 |
| **构建体积** | WASM约200KB+ gzipped |
| **维护状态** | Quickwit公司维护，非常活跃 |

**优势**: 性能最强，功能最全（类Lucene），中文分词完善
**劣势**: 体积最大，集成复杂度最高，前端WASM使用有门槛

## 4. 推荐方案

### 首选: FlexSearch（9.15分）

**推荐理由:**
- 性能最优：在JS搜索库中速度最快，适合10万+文档索引
- 体积最小：~15KB gzipped基础版，对前端加载几乎无影响
- 双环境支持：同时可在Node.js后端和浏览器前端使用，与Open-AwA的FastAPI+React架构完美匹配
- MIT许可证：商业友好，无版权顾虑
- 自定义encoder：可集成jieba分词器解决中文分词问题

### 备选: Orama（8.80分）

**推荐理由:**
- TypeScript原生：与Open-AwA前端技术栈（React+TS）高度一致
- 零依赖：不会引入额外的npm依赖链
- 混合搜索：支持全文+向量搜索，可与项目的向量存储（ChromaDB）互补
- 社区活跃：文档完善，更新频繁

### 组合方案（推荐）: FlexSearch + jieba分词

使用FlexSearch作为搜索引擎核心，通过自定义encoder集成jieba-js或nodejieba实现中文分词。前端使用FlexSearch的浏览器版本，后端（Python）使用Tantivy的Python绑定作为可选的高性能后端。
