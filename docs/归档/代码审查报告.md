## 二、严重问题 (P0 - 需立即修复)

### 2.1 [安全] Sandbox 命令注入漏洞

**文件**: `backend/security/sandbox.py` 第38行
**风险等级**: 严重

`execute_command` 方法使用 `asyncio.create_subprocess_shell` 直接执行传入的命令字符串，无任何输入校验或命令白名单过滤。攻击者若能控制 `command` 参数，可执行任意系统命令。

```python
# 当前危险实现
process = await asyncio.create_subprocess_shell(
    command,  # 未经任何校验
    stdout=asyncio.subprocess.PIPE,
    stderr=asyncio.subprocess.PIPE,
)
```

**改进建议**:
- 改用 `create_subprocess_exec` 并传入参数列表，禁止 shell 解释
- 实现命令白名单机制，参考 `skill_executor.py` 中的 `_execute_shell_action`
- 在执行前强制调用 `check_permission` 方法

### 2.2 [安全] 文件操作路径遍历漏洞

**文件**: `backend/security/sandbox.py` 第75-125行, `backend/skills/skill_executor.py` 第391-416行
**风险等级**: 严重

`Sandbox.execute_file_operation` 和 `SkillExecutor._execute_file_action` 均直接使用用户传入的 `file_path` 进行文件读写删除操作，未做任何路径校验。可被利用读取 `/etc/passwd`、`.env` 等敏感文件，或覆盖任意系统文件。

```python
# sandbox.py - 无路径校验
with open(file_path, 'r', encoding='utf-8') as f:
    content = f.read()
```

**改进建议**:
- 参考 `FileManagerSkill._validate_path` 实现统一的路径安全校验
- 限制文件操作在指定工作目录内（chroot 思路）
- 禁止 `..` 路径遍历和符号链接跟随

### 2.3 [安全] exec() 动态代码执行风险

**文件**: `backend/skills/skill_executor.py` 第37行
**风险等级**: 高

虽然实现了 `CodeValidator` 和受限 `__builtins__`，但存在以下绕过风险:
- `allowed_node_types` 包含 `FunctionDef`，可间接调用危险操作
- `getattr` 同时出现在 `dangerous_patterns` 和 `safe` 函数白名单中，逻辑矛盾
- 线程级超时机制无法真正终止正在运行的恶意代码

**改进建议**:
- 移除 `getattr` 在安全函数白名单中的条目
- 考虑使用进程级隔离替代线程级超时
- 收紧 `allowed_node_types`，移除 `FunctionDef` 或增加嵌套深度限制

### 2.4 [安全] Shell 命令白名单包含危险命令

**文件**: `backend/skills/skill_executor.py` 第445行
**风险等级**: 高

白名单中包含 `rm`、`chmod`、`chown`、`awk`、`sed`、`xargs` 等危险命令。`xargs` 可被利用执行白名单之外的任意命令。

**改进建议**:
- 移除 `rm`、`chmod`、`chown`、`xargs` 等高危命令
- 对 `awk`、`sed` 限制为只读操作
- 增加参数级别的校验（如禁止 `rm -rf /`）

## 三、高优先级问题 (P1 - 本迭代内修复)

### 3.1 [安全] Sandbox 权限检查未被调用

**文件**: `backend/security/sandbox.py`
**风险等级**: 中

`check_permission` 方法已实现但从未被 `execute_command` 和 `execute_file_operation` 调用，权限检查形同虚设。

**改进建议**: 在 `execute_command` 和 `execute_file_operation` 入口处强制调用 `check_permission`。

### 3.2 [安全] SQL 注入风险 - 迁移脚本

**文件**: `backend/migrate_db.py`
**风险等级**: 中

使用 f-string 拼接 DDL 语句。虽然当前变量来自硬编码列表，但模式本身不安全。

```python
cursor.execute(f"ALTER TABLE skills ADD COLUMN {column_name} {column_type}")
```

**改进建议**: 对变量做白名单校验，或使用 SQLAlchemy DDL API 替代原始 SQL。

### 3.3 [代码质量] 注释质量严重参差不齐

**涉及范围**: 全项目
**风险等级**: 中

大量函数的 docstring 为机械生成的模板文本，如:
```python
def verify_password(plain_password: str, hashed_password: str) -> bool:
    """
    处理verify、password相关逻辑，并为调用方返回对应结果。
    阅读时可结合入参、副作用与返回值理解它在整个链路中的定位。
    """
```

这类注释不仅没有提供有用信息，反而增加了代码噪音，降低了可读性。

**改进建议**:
- 清理所有模板化 docstring，替换为描述实际业务逻辑的注释
- 对公共 API 函数编写参数说明和返回值说明
- 对内部函数可省略 docstring，用行内注释解释关键逻辑

### 3.4 [测试] 后端核心模块测试覆盖率极低

**涉及范围**: `backend/core/`, `backend/api/routes/`, `backend/billing/`, `backend/security/`
**风险等级**: 高

后端约 48 个核心源文件缺少对应测试，包括:
- **core/**: agent.py, executor.py, planner.py, feedback.py, comprehension.py, model_service.py, metrics.py (全部无测试)
- **api/routes/**: 全部 10 个路由文件无测试
- **billing/**: 全部 7 个模块中仅 pricing_manager 有测试
- **security/**: sandbox.py, permission.py, audit.py (全部无测试)
- **memory/**: manager.py, experience_manager.py (全部无测试)

**改进建议**:
- 优先为 `core/agent.py`、`core/executor.py`、`api/routes/auth.py`、`api/routes/chat.py` 补充单元测试
- 为安全模块 `security/` 编写测试，确保权限检查和沙箱隔离逻辑正确
- 设定后端测试覆盖率目标（建议 >= 60%）

### 3.5 [前端] 开发环境自动注册登录硬编码凭据

**文件**: `frontend/src/App.tsx` 第89-121行
**风险等级**: 中

开发模式下自动使用硬编码的 `test_user_default` / `test_password_123` 注册并登录。如果构建配置错误导致生产环境也启用了 DEV 模式，将产生安全风险。

```typescript
if (import.meta.env.DEV) {
  const testUser = {
    username: 'test_user_default',
    password: 'test_password_123'
  }
  // ...自动注册并登录
}
```

**改进建议**:
- 增加额外的环境变量开关控制自动登录行为
- 在生产构建中通过 tree-shaking 确保此代码被完全移除
- 考虑将测试凭据移至 `.env.development` 文件

## 四、中优先级问题 (P2 - 下个迭代修复)

### 4.1 [性能] 后端缺少数据库连接池配置

**文件**: `backend/db/models.py`

SQLAlchemy engine 创建时未配置连接池参数，使用默认的 `QueuePool`。对于 SQLite 这不是大问题，但如果迁移到 PostgreSQL 等数据库，需要显式配置。

```python
engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {}
    # 缺少 pool_size, max_overflow, pool_recycle 等配置
)
```

**改进建议**: 增加 `pool_size=5, max_overflow=10, pool_recycle=3600` 等配置。

### 4.2 [性能] 日志缓冲区使用内存 deque

**文件**: `backend/config/logging.py`

`_LOG_BUFFER = deque(maxlen=5000)` 将日志存储在内存中，查询时遍历全部条目。在高并发场景下可能成为性能瓶颈。

**改进建议**:
- 考虑将日志持久化到数据库或文件
- 查询接口增加索引支持
- 或使用专业的日志聚合服务（如 ELK）

### 4.3 [依赖] 后端 requirements.txt 缺少版本上限

**文件**: `backend/requirements.txt`

部分依赖使用 `>=` 而非精确版本或范围约束:
```
fastapi>=0.109.1
python-jose[cryptography]>=3.4.0
python-multipart>=0.0.22
click>=8.1.0
pytest-asyncio>=0.23.0
```

**改进建议**: 使用 `~=` 或 `>=x.y,<x+1.0` 格式限制主版本，避免不兼容升级。

### 4.4 [依赖] pytest-asyncio 不应出现在 requirements.txt

**文件**: `backend/requirements.txt`

`pytest-asyncio` 是测试依赖，不应包含在生产依赖文件中。

**改进建议**: 创建 `requirements-dev.txt` 分离开发/测试依赖。

### 4.5 [代码质量] 前端 ESLint 关闭了关键规则

**文件**: `frontend/eslint.config.js`

```javascript
'@typescript-eslint/no-explicit-any': 'off',
```

关闭 `no-explicit-any` 导致项目中大量使用 `any` 类型，削弱了 TypeScript 的类型安全优势。

**改进建议**: 逐步启用此规则（先设为 `warn`），消除现有 `any` 使用。

### 4.6 [代码质量] 前端 API 层类型定义不严格

**文件**: `frontend/src/shared/api/api.ts`

多个 API 方法使用 `any` 类型:
```typescript
install: (skill: any) => api.post('/skills', skill),
create: (prompt: any) => api.post('/prompts', prompt),
update: (id: string, prompt: any) => api.put(`/prompts/${id}`, prompt),
```

**改进建议**: 为每个 API 方法定义明确的请求/响应类型接口。

### 4.7 [代码质量] 前端测试文件命名不规范

**文件**: `frontend/src/__tests__/`

存在大量 `.test.test.tsx` 双重后缀的测试文件:
- `App.test.test.tsx`
- `features_billing_BillingPage.test.test.tsx`
- `features_chat_ChatPage.test.test.tsx`
- 等约 20 个文件

**改进建议**: 统一测试文件命名为 `*.test.tsx` 或 `*.spec.tsx`。

### 4.8 [结构] 前端覆盖率配置仅包含两个文件

**文件**: `frontend/vitest.config.ts`

```typescript
coverage: {
  include: [
    'src/features/plugins/PluginDebugPanel.tsx',
    'src/shared/utils/logger.ts'
  ],
}
```

覆盖率统计仅包含 2 个文件，无法反映真实的项目覆盖率。

**改进建议**: 将 `include` 改为 `['src/**/*.{ts,tsx}']`，排除测试文件和类型声明。

## 七、最佳实践建议

### 7.1 安全加固 (最高优先级)

1. **重写 Sandbox 模块**: 当前沙箱实现存在根本性安全缺陷，建议基于进程隔离（如 Docker 容器或 nsjail）重新设计
2. **统一路径校验**: 提取 `FileManagerSkill._validate_path` 为公共工具函数，所有文件操作必须经过校验
3. **实施 RBAC**: 当前权限系统仅区分 admin/user 两个角色，建议实现细粒度的基于角色的访问控制
4. **增加速率限制**: 登录接口和 API 接口缺少速率限制，容易被暴力破解或 DDoS
5. **JWT Token 刷新机制**: 当前 Token 有效期 24 小时且无刷新机制，建议实现 refresh token

### 7.2 代码质量提升

1. **清理模板化注释**: 全项目搜索并替换 "处理xxx相关逻辑" 等模板文本
2. **消除 any 类型**: 前端逐步启用 `no-explicit-any` 规则
3. **统一错误处理**: 后端建立统一的异常层次结构和错误码体系
4. **引入 Python 类型检查**: 配置 mypy 或 pyright，逐步增加类型注解

### 7.3 测试体系建设

1. **后端测试优先级**: core/agent.py > api/routes/auth.py > api/routes/chat.py > security/ > billing/
2. **集成测试**: 增加 API 级别的集成测试，使用 TestClient
3. **前端覆盖率配置**: 扩大覆盖率统计范围至全部源文件
4. **测试文件命名规范化**: 清理 `.test.test.tsx` 双重后缀

### 7.4 架构优化

1. **数据库迁移工具**: 引入 Alembic 替代手写迁移脚本
2. **配置管理**: 分离开发/测试/生产配置
3. **容器化**: 编写 Dockerfile 和 docker-compose.yml
4. **API 版本管理**: 当前所有路由挂载在 `/api` 下，建议增加版本前缀 `/api/v1`
