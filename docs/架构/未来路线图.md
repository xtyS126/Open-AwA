## 1. MCP 协议支持

### 1.1 当前状态

- 完成度: **90%**
- MCP 客户端 (`client.py`) 已实现，支持 stdio 和 SSE 传输
- MCP 管理器 (`manager.py`) 为线程安全单例，支持多服务器生命周期管理
- 协议层 (`protocol.py`) 完整实现 JSON-RPC 2.0 消息格式
- API 路由 (`api/routes/mcp.py`) 提供 9 个端点：服务器 CRUD、连接管理、工具发现与调用、配置快照与回滚
- 前端 `MCPSettings.tsx` 已完成服务器配置管理 UI
- agent 主流程已集成 MCP 工具（`mcp_{server_id}/{tool_name}` 分发）

### 1.2 待完成

#### 1.2.1 资源读取支持

- 实现 MCP `resources/list` 和 `resources/read` 协议
- Executor 层增加 MCP 资源调用能力

#### 1.2.2 SSE 安全增强

- SSE 端点增加 origin 校验
- 连接认证令牌机制

#### 1.2.3 前端增强

- MCP 工具调用结果展示 UI
- 工具调用历史记录

### 1.3 验收标准

- [x] 通过 stdio 连接本地 MCP Server
- [x] 通过 SSE 连接远程 MCP Server
- [x] 工具发现正确解析 tools/list 响应

- [x] Agent 主流程可自动选择并调用 MCP 工具
- [x] 前端可配置/管理 MCP Server

- [ ] 支持 MCP resources/list 和 resources/read
- [ ] SSE 端点 origin 校验

## 2. SubAgent 子智能体

### 2.1 当前状态

- 完成度: **70%**
- `subagent.py` 已实现完整的 StateGraph 执行引擎
- 支持 DAG 节点定义、条件路由、重试、超时、并行执行
- API 路由已注册 (`api/routes/subagents.py`)
- 缺具体 Agent 节点实现和前端界面

### 2.2 目标

实现可配置的子智能体系统，用户可定义多个专业 Agent 并编排其协作流程。

### 2.3 实现计划

#### 2.3.1 内置 Agent 节点实现

- 代码审查 Agent（CodeReviewAgent）
- 搜索 Agent（SearchAgent）
- 数据分析 Agent（DataAnalysisAgent）
- 每个 Agent 注册到 SubAgentManager

#### 2.3.2 图定义持久化

- 数据库表 `subagent_definitions` 存储图定义
- API 端点支持图的 CRUD

#### 2.3.3 前端界面

- SubAgent 管理页面：图定义编辑器（可视化 DAG）
- 执行监控：实时查看节点执行状态和结果
- 执行历史记录

### 2.4 验收标准

- [x] StateGraph 引擎支持 DAG 执行
- [x] 并行和串行执行模式
- [ ] 至少 3 个内置 Agent 节点可用
- [ ] 图定义可持久化和加载
- [ ] 前端可创建和编辑 Agent 图
- [ ] 执行结果可实时查看

## 4. 工作流引擎

### 4.1 当前状态

- 完成度: **80%**
- YAML/JSON 定义解析 (`parser.py`)
- 支持 4 种步骤类型：tool（内置工具）、skill（技能）、plugin（插件）、condition（条件分支）
- 模板渲染引擎：`{{ context.x }}`、`{{ steps.y }}`、`{{ last_result.z }}`
- AST 安全条件表达式求值
- 执行记录入库，支持错误处理（continue/stop）
- 缺并行步骤执行和工作流可视化编辑器

### 4.2 目标

增强工作流的执行能力和用户体验，支持并行执行、子工作流组合和可视化编辑。

### 4.3 实现计划

#### 4.3.1 并行执行

- 无依赖步骤自动并行执行
- `parallel` 步骤类型显式定义并行组
- 并行结果聚合

#### 4.3.2 子工作流组合

- `sub_workflow` 步骤类型引用其他工作流
- 参数传递与结果回传
- 递归深度限制

#### 4.3.3 前端可视化编辑器

- 拖拽式工作流编辑器
- 步骤节点连线
- 实时预览执行流

### 4.4 验收标准

- [x] 条件分支步骤正确执行

## 5. Chain-of-Thought 改进

### 5.1 当前状态

- 完成度: **85%**
- 多厂商推理内容解析：OpenAI o-series (`reasoning_effort`)、Anthropic Claude (`thinking budget_tokens`)、DeepSeek R1、Zhipu GLM
- SSE/WebSocket 流式传输推理内容（`event: reasoning`）
- 前端 `ReasoningContent.tsx` 支持折叠/展开、流式跟踪
- `build_thinking_params()` 将 0-5 深度映射到各厂商参数
- Token 统计已区分 reasoning tokens 和 output tokens

### 5.2 目标

完善 CoT 的高级功能，增加深度自动调节和审计能力。

### 5.3 实现计划

#### 5.3.1 深度自动调节

- 根据问题复杂度自动选择推理深度
- 简单问候/事实查询 -> 低深度，数学/编程 -> 高深度
- 用户可覆盖自动选择

#### 5.3.2 推理审计

- 推理过程存储到会话记录
- 推理耗时统计和展示
- 推理内容导出功能

### 5.4 验收标准

- [x] 支持 OpenAI/Anthropic/DeepSeek/Zhipu 的推理内容解析
- [x] SSE 流式传输包含 reasoning 类型消息
- [x] 前端正确展示推理过程内容
- [x] reasoning_effort 参数可配置并生效
- [x] Token 统计区分推理和输出
- [ ] 问题复杂度自动匹配推理深度
- [ ] 推理内容可导出

## 7. 安全增强

## 实施优先级

| 优先级 | 模块 | 理由 |
|--------|------|------|
| P0 | SubAgent 子智能体 | 核心差异化能力，Agent 编排基础 |
| P0 | 工作流引擎（并行+可视化） | 自动化场景核心需求 |
| P1 | 计费系统完善 | 用量预警直接影响用户体验 |
| P1 | MCP 协议（资源支持） | 生态扩展的关键缺口 |
| P1 | 微信多媒体消息 | 已有用户群的刚需 |
| P2 | 插件市场（远端下载） | 社区生态建设的基础 |
| P2 | 多语言模型（故障转移） | 提升服务可用性 |
| P2 | 安全增强（细粒度权限） | 生产环境部署前置 |
| P3 | Chain-of-Thought（自动深度） | 体验优化，按需推进 |
| P3 | 前端 UI（响应式/i18n） | 持续优化项 |
